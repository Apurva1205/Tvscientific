2) Supervised vs. Unsupervised learning

Explanation. Supervised learning optimizes a loss against known targets (e.g., cross‑entropy for conversions). Unsupervised learning discovers structure in unlabeled data (clustering, density modeling, dimensionality reduction) and can feed supervised tasks via features or pre‑training.

Example (ad tech). Use supervised models to predict the probability of a site visitor converting after a TV ad impression; use unsupervised clustering on engagement signals to discover high‑value audience segments and feed those segment IDs as features to the converter.

How I’d approach. Start with a supervised baseline (regularized logistic/GBM) on clean labels; in parallel, run K‑Means/HDBSCAN on behavior features (recency/frequency, dwell time) to create segments; evaluate whether adding segment features improves AUC‑PR/CPA; iterate by pretraining embeddings (autoencoder/SimCLR‑style) if sparse signals dominate.

3) How a decision tree works

Explanation. A tree does greedy, top‑down splits that maximize impurity reduction (Gini/entropy). Leaves hold predictions; depth/leaf size/pruning regularize variance.

Example. For conversion prediction, early splits might be on "minutes since ad exposure" and "device type", capturing non‑linear interactions. A small depth avoids memorizing campaign IDs.

How I’d approach. Start with gradient‑boosted trees (LightGBM/XGBoost) with monotonic constraints if needed; perform stratified CV; tune learning rate, num_leaves, min_data_in_leaf; add calibrated probabilities (isotonic/Platt) and evaluate business metrics (lift @ spend, CPA).

4) Handling imbalanced datasets

Explanation. Rare positives bias accuracy‑style metrics. Use resampling (SMOTE/undersample), re‑weighting, focal loss, and threshold tuning with cost‑aware metrics.

Example. Only 0.5% viewers convert after a TV ad. I train a GBM with class weights, evaluate on AUC‑PR and recall@K, then adjust thresholds to meet CPA targets.

How I’d approach. Establish a plain logistic baseline and a GBM with weights; add domain features (time‑since‑ad, channel frequency caps); compare SMOTE vs. weights; calibrate scores; choose an operating point using a profit curve; lock with stratified/group CV to prevent leakage across campaigns.

5) Handling missing data

Explanation. Identify MCAR/MAR/MNAR. Impute with robust stats (median/mode), model‑based methods, and add missingness indicators when predictive. Prefer algorithms tolerant to NAs (trees).

Example. Device type is missing for 12% of events. I encode a missing flag, impute with mode for linear models, and compare against a tree method that natively handles NAs.

How I’d approach. Profile missingness by feature × segment; try simple imputations first; run sensitivity analysis (multiple imputations); select approach that minimizes downstream metric loss while keeping pipelines simple and explainable.

6) Preventing overfitting

Explanation. Control model capacity and validate properly. Techniques: regularization, early stopping, dropout (NNs), pruning (trees), feature curation, and leakage checks.

Example. A deep net overfits sparse view events. Switching to GBM with early stopping and stronger feature selection improves PR‑AUC and generalization.

How I’d approach. Use nested CV; set conservative early‑stopping; remove leaky features (post‑event signals); add noise/augmentation when appropriate; conduct ablation to keep only features that add stable lift across folds.

8) Challenges & remedies

Non‑stationarity & drift (Energy forecasting). Demand shifted with holidays, weather spikes, and DST. Remedies: added calendar/weather/exogenous features, per‑region models, time‑based CV, PSI/KS feature drift alerts, and weekly retraining—helped drive the ~8–12% MAPE improvement already noted.

Delayed & noisy labels (Ads/Resume NLP). Conversions arrive late and resume–JD “fit” labels are subjective. Remedies: used time‑lagged labels with right‑censoring windows, time‑ordered validation, and “silver” labels from heuristic matchers; spot‑checked samples and smoothed training with label‑quality weights.

Class imbalance & sparsity. Positives <1% for conversions; long‑tail resumes/skills. Remedies: class weights/focal loss, PR‑AUC/recall@K as primary metrics, threshold tuning to CPA/precision targets, top‑K ranking for retrieval, and isotonic calibration for reliable scores.

Data quality & schema drift. Upstream changed device enums / missing cols broke pipelines. Remedies: Great Expectations + Pydantic schema checks, freshness SLAs, and data contracts; fail‑fast with alerts; versioned datasets and backfills for safe re‑training.

Latency & cost at serving. Early prototypes had heavy Python preprocessing. Remedies: vectorized feature prep, LightGBM for tabular tasks, cached features, and containerized FastAPI with autoscaling; reduced p95 latency while keeping throughput budget‑friendly.

Explainability & stakeholder alignment. Hard to tie model lift to outcomes. Remedies: SHAP summaries + feature cards, dashboards that map MAE/MAPE to $ impact (ROAS/CPA proxies), and weekly readouts to prioritize next experiments.
